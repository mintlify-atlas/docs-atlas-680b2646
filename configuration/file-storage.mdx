# File Storage Configuration

LibreChat supports multiple file storage strategies for avatars, images, and documents. You can use a single strategy for all files or configure granular strategies for different file types.

## Storage Strategies

LibreChat supports four storage backends:

<CardGroup cols={2}>
  <Card title="Local Storage" icon="hard-drive">
    Store files on the server's local filesystem. Simple but not suitable for distributed deployments.
  </Card>
  <Card title="AWS S3" icon="aws">
    Store files in Amazon S3 or S3-compatible services (MinIO, Hetzner, Backblaze B2).
  </Card>
  <Card title="Firebase Storage" icon="fire">
    Use Google Firebase Cloud Storage with automatic optimization.
  </Card>
  <Card title="Azure Blob Storage" icon="cloud">
    Store files in Microsoft Azure Blob Storage.
  </Card>
</CardGroup>

## Configuration Methods

<Tabs>
  <Tab title="Single Strategy (Legacy)">
    Use one storage strategy for all file types:
    
    ```yaml librechat.yaml
    fileStrategy: "s3"
    ```
    
    Options: `"local"`, `"s3"`, `"firebase"`, `"azure_blob"`
  </Tab>
  
  <Tab title="Granular Strategies (Recommended)">
    Configure different strategies for different file types:
    
    ```yaml librechat.yaml
    fileStrategy:
      avatar: "s3"        # User/agent avatars
      image: "firebase"   # Chat images
      document: "local"   # Document uploads
    ```
    
    **File Types**:
    - `avatar`: User and agent avatar images
    - `image`: Images uploaded in chats or generated by AI
    - `document`: Document uploads (PDFs, text files, etc.)
  </Tab>
</Tabs>

<Tip>
  **Why use granular strategies?**
  
  Mix and match based on your needs:
  - Use **S3** for avatars for fast global CDN access
  - Use **Firebase** for images with automatic optimization and resizing
  - Use **local** for documents for privacy/compliance requirements
</Tip>

## Local Storage

Stores files on the server's filesystem.

### Configuration

```yaml librechat.yaml
fileStrategy: "local"
# Or granular:
fileStrategy:
  avatar: "local"
  image: "local"
  document: "local"
```

### File Locations

Files are stored in the following directories:

```bash
/path/to/librechat/
  client/
    public/
      images/           # Generated images
  api/
    data/
      uploads/          # Document uploads
      avatars/          # User/agent avatars
```

### Pros and Cons

<CardGroup cols={2}>
  <Card title="Advantages" icon="check">
    - No external dependencies
    - Simple setup
    - Free
    - Fast for single-server deployments
    - Full data control
  </Card>
  <Card title="Disadvantages" icon="xmark">
    - Not suitable for distributed/scaled deployments
    - No automatic backups
    - Storage limited by server disk space
    - Files lost if container/server is destroyed
  </Card>
</CardGroup>

<Warning>
  **Production Considerations**:
  
  For Docker deployments, mount volumes to persist files:
  
  ```yaml docker-compose.yml
  volumes:
    - ./uploads:/app/api/data/uploads
    - ./avatars:/app/api/data/avatars
    - ./images:/app/client/public/images
  ```
</Warning>

## AWS S3

Store files in Amazon S3 or S3-compatible services.

### Environment Variables

<ParamField path="AWS_ACCESS_KEY_ID" type="string" required>
  AWS access key ID
  
  ```bash
  AWS_ACCESS_KEY_ID=AKIAXXXXX
  ```
</ParamField>

<ParamField path="AWS_SECRET_ACCESS_KEY" type="string" required>
  AWS secret access key
  
  ```bash
  AWS_SECRET_ACCESS_KEY=xxxxx
  ```
</ParamField>

<ParamField path="AWS_REGION" type="string" required>
  AWS region for the S3 bucket
  
  ```bash
  AWS_REGION=us-east-1
  ```
</ParamField>

<ParamField path="AWS_BUCKET_NAME" type="string" required>
  S3 bucket name
  
  ```bash
  AWS_BUCKET_NAME=librechat-files
  ```
</ParamField>

<ParamField path="AWS_ENDPOINT_URL" type="string">
  Custom S3 endpoint URL (for S3-compatible services)
  
  ```bash
  # MinIO:
  AWS_ENDPOINT_URL=https://minio.example.com
  
  # Hetzner:
  AWS_ENDPOINT_URL=https://fsn1.your-objectstorage.com
  
  # Backblaze B2:
  AWS_ENDPOINT_URL=https://s3.us-west-002.backblazeb2.com
  
  # DigitalOcean Spaces:
  AWS_ENDPOINT_URL=https://nyc3.digitaloceanspaces.com
  ```
</ParamField>

<ParamField path="AWS_FORCE_PATH_STYLE" type="boolean" default="false">
  Use path-style URLs instead of virtual-hosted-style
  
  <Note>
    Required for S3-compatible providers (MinIO, Hetzner, Backblaze B2, etc.) that don't support virtual-hosted-style URLs. Not needed for AWS S3.
  </Note>
  
  ```bash
  # For MinIO, Hetzner, Backblaze, etc.:
  AWS_FORCE_PATH_STYLE=true
  
  # AWS S3:
  AWS_FORCE_PATH_STYLE=false
  ```
</ParamField>

### YAML Configuration

```yaml librechat.yaml
fileStrategy: "s3"
# Or granular:
fileStrategy:
  avatar: "s3"
  image: "s3"
  document: "s3"
```

### S3 Bucket Setup

<Steps>
  <Step title="Create S3 Bucket">
    Create a new S3 bucket in your AWS account or S3-compatible service
  </Step>
  
  <Step title="Configure CORS">
    Add CORS policy to allow uploads from your domain:
    
    ```json
    [
      {
        "AllowedHeaders": ["*"],
        "AllowedMethods": ["GET", "PUT", "POST", "DELETE"],
        "AllowedOrigins": ["https://yourdomain.com"],
        "ExposeHeaders": ["ETag"]
      }
    ]
    ```
  </Step>
  
  <Step title="Set Bucket Policy">
    Configure bucket policy for public read access (if needed):
    
    ```json
    {
      "Version": "2012-10-17",
      "Statement": [
        {
          "Sid": "PublicReadGetObject",
          "Effect": "Allow",
          "Principal": "*",
          "Action": "s3:GetObject",
          "Resource": "arn:aws:s3:::librechat-files/*"
        }
      ]
    }
    ```
  </Step>
  
  <Step title="Create IAM User (AWS)">
    For AWS S3, create an IAM user with S3 permissions:
    
    ```json
    {
      "Version": "2012-10-17",
      "Statement": [
        {
          "Effect": "Allow",
          "Action": [
            "s3:PutObject",
            "s3:GetObject",
            "s3:DeleteObject",
            "s3:ListBucket"
          ],
          "Resource": [
            "arn:aws:s3:::librechat-files",
            "arn:aws:s3:::librechat-files/*"
          ]
        }
      ]
    }
    ```
  </Step>
</Steps>

## Firebase Storage

Use Google Firebase Cloud Storage with automatic optimization.

### Environment Variables

<ParamField path="FIREBASE_API_KEY" type="string" required>
  Firebase API key
  
  ```bash
  FIREBASE_API_KEY=AIzaSyXXX
  ```
</ParamField>

<ParamField path="FIREBASE_AUTH_DOMAIN" type="string" required>
  Firebase auth domain
  
  ```bash
  FIREBASE_AUTH_DOMAIN=your-project.firebaseapp.com
  ```
</ParamField>

<ParamField path="FIREBASE_PROJECT_ID" type="string" required>
  Firebase project ID
  
  ```bash
  FIREBASE_PROJECT_ID=your-project-id
  ```
</ParamField>

<ParamField path="FIREBASE_STORAGE_BUCKET" type="string" required>
  Firebase storage bucket name
  
  ```bash
  FIREBASE_STORAGE_BUCKET=your-project.appspot.com
  ```
</ParamField>

<ParamField path="FIREBASE_MESSAGING_SENDER_ID" type="string" required>
  Firebase messaging sender ID
  
  ```bash
  FIREBASE_MESSAGING_SENDER_ID=123456789012
  ```
</ParamField>

<ParamField path="FIREBASE_APP_ID" type="string" required>
  Firebase app ID
  
  ```bash
  FIREBASE_APP_ID=1:123456789012:web:xxxxx
  ```
</ParamField>

### YAML Configuration

```yaml librechat.yaml
fileStrategy: "firebase"
# Or granular:
fileStrategy:
  avatar: "firebase"
  image: "firebase"
  document: "firebase"
```

### Firebase Setup

<Steps>
  <Step title="Create Firebase Project">
    1. Go to [Firebase Console](https://console.firebase.google.com/)
    2. Create a new project
    3. Enable Firebase Storage
  </Step>
  
  <Step title="Get Configuration">
    1. Go to Project Settings
    2. Scroll to "Your apps" section
    3. Click "Web app" and copy the configuration values
  </Step>
  
  <Step title="Configure Storage Rules">
    Set up Firebase Storage security rules:
    
    ```javascript
    rules_version = '2';
    service firebase.storage {
      match /b/{bucket}/o {
        match /{allPaths=**} {
          allow read: if true;
          allow write: if request.auth != null;
        }
      }
    }
    ```
  </Step>
</Steps>

### Features

- Automatic image optimization and resizing
- CDN delivery for fast global access
- Built-in security rules
- Generous free tier (5GB storage, 1GB/day downloads)
- Automatic backups

## Azure Blob Storage

Store files in Microsoft Azure Blob Storage.

### Environment Variables

<ParamField path="AZURE_STORAGE_CONNECTION_STRING" type="string" required>
  Azure storage account connection string
  
  ```bash
  AZURE_STORAGE_CONNECTION_STRING="DefaultEndpointsProtocol=https;AccountName=myaccount;AccountKey=xxxxx;EndpointSuffix=core.windows.net"
  ```
</ParamField>

<ParamField path="AZURE_CONTAINER_NAME" type="string" default="files">
  Azure blob container name
  
  ```bash
  AZURE_CONTAINER_NAME=files
  ```
</ParamField>

<ParamField path="AZURE_STORAGE_PUBLIC_ACCESS" type="boolean" default="false">
  Enable public access to blobs
  
  ```bash
  AZURE_STORAGE_PUBLIC_ACCESS=false
  ```
</ParamField>

### YAML Configuration

```yaml librechat.yaml
fileStrategy: "azure_blob"
# Or granular:
fileStrategy:
  avatar: "azure_blob"
  image: "azure_blob"
  document: "azure_blob"
```

### Azure Setup

<Steps>
  <Step title="Create Storage Account">
    1. Go to [Azure Portal](https://portal.azure.com/)
    2. Create a new Storage Account
    3. Choose performance and replication options
  </Step>
  
  <Step title="Create Container">
    1. In your storage account, go to **Containers**
    2. Create a new container named `files`
    3. Set access level (Private or Blob)
  </Step>
  
  <Step title="Get Connection String">
    1. Go to **Access keys** in your storage account
    2. Copy the connection string from key1 or key2
  </Step>
  
  <Step title="Configure CORS">
    Add CORS rules for your domain:
    
    ```xml
    <CorsRule>
      <AllowedOrigins>https://yourdomain.com</AllowedOrigins>
      <AllowedMethods>GET,PUT,POST,DELETE</AllowedMethods>
      <AllowedHeaders>*</AllowedHeaders>
      <ExposedHeaders>*</ExposedHeaders>
      <MaxAgeInSeconds>3600</MaxAgeInSeconds>
    </CorsRule>
    ```
  </Step>
</Steps>

## File Upload Configuration

Configure file upload limits per endpoint in `librechat.yaml`:

```yaml librechat.yaml
fileConfig:
  endpoints:
    assistants:
      fileLimit: 5                    # Max number of files
      fileSizeLimit: 10               # Max size per file (MB)
      totalSizeLimit: 50              # Max total size (MB)
      supportedMimeTypes:
        - "image/.*"
        - "application/pdf"
    
    openAI:
      disabled: true                  # Disable file uploads
    
    default:
      totalSizeLimit: 20
    
    YourCustomEndpointName:
      fileLimit: 2
      fileSizeLimit: 5
  
  # Global settings
  serverFileSizeLimit: 100            # Global server limit (MB)
  avatarSizeLimit: 2                  # Avatar limit (MB)
  
  # Image generation sizing
  imageGeneration:
    percentage: 100                   # Percentage-based sizing
    # OR
    # px: 1024                        # Pixel-based sizing
  
  # Client-side image resizing
  clientImageResize:
    enabled: false                    # Enable client-side resizing
    maxWidth: 1900                    # Max width (px)
    maxHeight: 1900                   # Max height (px)
    quality: 0.92                     # JPEG quality (0.0-1.0)
```

## Migrating Between Storage Strategies

When changing storage strategies, existing files won't be automatically migrated.

<Warning>
  **Migration Considerations**:
  
  - Files uploaded with one strategy remain in that storage location
  - Users may see broken images/avatars after switching strategies
  - Manual migration is required to move existing files
</Warning>

### Migration Steps

<Steps>
  <Step title="Set New Strategy">
    Update `fileStrategy` in `librechat.yaml` to your new storage backend
  </Step>
  
  <Step title="Export Existing Files">
    Download all files from your current storage:
    
    ```bash
    # For local storage:
    tar -czf files-backup.tar.gz \
      ./api/data/uploads \
      ./api/data/avatars \
      ./client/public/images
    
    # For S3:
    aws s3 sync s3://your-bucket ./files-backup
    ```
  </Step>
  
  <Step title="Upload to New Storage">
    Upload files to your new storage backend using its API or CLI
  </Step>
  
  <Step title="Update Database References">
    Update file references in MongoDB if paths have changed
  </Step>
</Steps>

## Performance Optimization

### CDN Integration

For production deployments, consider using a CDN:

<Tabs>
  <Tab title="CloudFront (S3)">
    1. Create CloudFront distribution pointing to S3 bucket
    2. Configure origin access identity
    3. Update S3 bucket policy
    4. Use CloudFront URL for file access
  </Tab>
  
  <Tab title="Azure CDN">
    1. Create Azure CDN endpoint
    2. Point to your blob storage container
    3. Configure caching rules
    4. Use CDN URL for file access
  </Tab>
  
  <Tab title="Firebase Hosting">
    Firebase Storage automatically uses CDN for global delivery
  </Tab>
</Tabs>

### Caching Headers

Configure static file caching:

```bash .env
# Cache static files for 2 days
STATIC_CACHE_MAX_AGE=172800

# CDN cache for 1 day
STATIC_CACHE_S_MAX_AGE=86400
```

### Image Optimization

Enable client-side image resizing to reduce upload sizes:

```yaml librechat.yaml
fileConfig:
  clientImageResize:
    enabled: true
    maxWidth: 1900
    maxHeight: 1900
    quality: 0.92
```

## Comparison

| Feature | Local | S3 | Firebase | Azure Blob |
|---------|-------|-----|----------|------------|
| **Setup Complexity** | Easy | Medium | Medium | Medium |
| **Cost** | Free | Pay-per-use | Free tier + pay | Pay-per-use |
| **Scalability** | Limited | Excellent | Excellent | Excellent |
| **CDN** | No | Optional | Built-in | Optional |
| **Automatic Optimization** | No | No | Yes | No |
| **Best For** | Development | Production | Production | Enterprise |
| **Distributed Deployments** | No | Yes | Yes | Yes |

## Troubleshooting

<AccordionGroup>
  <Accordion title="Files not uploading">
    - Verify storage credentials are correct
    - Check bucket/container permissions
    - Ensure CORS is configured correctly
    - Review file size limits
    - Check network connectivity to storage service
  </Accordion>
  
  <Accordion title="Images not loading">
    - Verify files were uploaded successfully
    - Check bucket/container public access settings
    - Ensure CORS allows your domain
    - Verify file URLs are accessible
    - Check browser console for CORS errors
  </Accordion>
  
  <Accordion title="S3-compatible service issues">
    - Set `AWS_FORCE_PATH_STYLE=true` for most S3-compatible services
    - Verify endpoint URL is correct
    - Check if service supports virtual-hosted-style URLs
    - Test credentials with S3 CLI tools
  </Accordion>
  
  <Accordion title="Performance issues">
    - Enable CDN for faster delivery
    - Configure caching headers
    - Enable client-side image resizing
    - Consider using granular strategies
    - Review file size limits
  </Accordion>
</AccordionGroup>
